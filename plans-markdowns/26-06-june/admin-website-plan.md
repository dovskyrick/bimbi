# Admin Website — Feasibility & Design Notes

## What You're Describing

A read-only mirror of the public gallery UI, repurposed as an edit surface. Images open a file-upload modal on click. Text fields (title, price, medium, dimensions, year, description) become inline-editable on click. A single **Save** button in the top-right commits all staged changes to Firestore and Firebase Storage.

---

## Is It Doable?

Yes, completely. Nothing here is technically novel. It is a thin layer on top of what already exists — same components, same data model, same Firebase project. The scope is well-contained because you are explicitly **not** adding new paintings, only editing existing ones.

---

## Difficulty: 5 / 10

The UI recycling and Firestore update calls are easy (2/10). The friction points that push it to a 5 are:

- **Image replacement** — uploading a new file to Storage, deleting or overwriting the old one, regenerating the thumbnail, and writing the new URLs back to Firestore requires care to keep things atomic and not leave orphaned files.
- **Staging unsaved changes** — holding edits in local state across multiple fields and images before a single Save press needs a clean state pattern or you will have subtle bugs.
- **Thumbnail regeneration** — the current thumbnail pipeline lives in the Node CLI script using Sharp, which does not run in a browser. Either the browser skips thumbnails and only updates the full image, or a small Cloud Function or local proxy handles resize on upload.

---

## Steps to Build It

1. Scaffold `admin-web/` as a second Vite + React app inside the monorepo, sharing the same `firebase.ts` config.
2. Copy and adapt the gallery components into editable versions.
3. Implement inline text editing per field with a dirty-state tracker.
4. Implement image-click → file picker → Storage upload → URL update in Firestore.
5. Wire the Save button to flush all staged changes in a Firestore batch write.

---

## Localhost-Only: Perfectly Fine

Since this never gets deployed, there is no public attack surface. Running on `localhost:5174` (or similar) is entirely satisfying for this purpose. No HTTPS, no hosting config needed.

---

## Authentication

The current upload CLI authenticates via a **service account key** (`service-account.json`) which bypasses Firestore security rules entirely — that is why writes work from the terminal. The public Firestore rules deny all client writes.

For the localhost admin site, you have two options:

- **Option A (recommended, simplest):** Temporarily loosen the Firestore and Storage rules to allow writes from an authenticated Firebase user, add Firebase Auth (Google sign-in, one click), and only allow writes if the signed-in email matches yours. This takes about 30 minutes and is the cleanest approach.
- **Option B (no auth at all):** Open the rules wide while developing locally, close them when done. This works but is risky if you forget.

Option A is the right call. It also gives the admin site a real login gate, which matters if the localhost port is ever accidentally exposed on a shared network.

---

## Summary

This is a well-scoped, satisfying task. Two to three focused sessions would cover it.

**On thumbnails:** the current live site only uses `imageUrl` (full-size). `thumbnailUrl` is generated by the CLI and stored in Firestore, but only referenced in `Gallery.tsx` — a grid-layout page that exists in the code but is not wired up to the app. For the admin site, thumbnail regeneration is a non-issue: only `imageUrl` needs updating when an image is replaced. The thumbnail concern can be revisited if and when the grid gallery is ever activated.
